useThreadStore.ts – Zustand Store for Threads
The store manages:
List of threads (threads), the current thread (currentThread), loading states, and errors.
Replies (comments) per thread.
Optimistic UI updates for adding replies and marking best answers.
Key methods:
fetchThreads(page): Fetches paginated thread list.
fetchThreadBySlug(slug): Fetches a single thread + its comments.
setReplies(slug, replies): Updates the reply tree, marking best answer.
addReply(slug, reply): Adds a new reply, correctly nested if parentId exists.
refetchReplies(slug): Refetches comments to sync with backend.
markBestReply(slug, commentId): Marks a comment as best, reorders tree to place it at top.
reset(): Clears store.
Recursive Helpers:
insertReply: Inserts a reply at the correct parent in nested children.
markBestInTree: Recursively marks the best comment in the tree.


4️⃣ Workflow Summary
Adding a reply:
User types reply → handleReplySubmit.
POST request sent to /api/threads/${slug}/comments.
On success, addReply() updates Zustand store.
UI shows reply immediately (optimistic UI).
Voting:
User clicks vote → handleVote(type).
Update upvotes/downvotes immediately.
POST request sent to /api/comments/${id}/vote.
On failure, revert the vote counts.
Marking Best Answer:
Only thread author sees button.
Click → handleAcceptBest.
Update UI optimistically via markBestReply.
POST request sent to /api/comments/${id}/accept.
Refetch replies to ensure consistency.
Nested Replies:
Recursive component rendering: children are displayed with increasing marginLeft to indicate nesting

User clicks "Flag"

Increment Redis counter
If flags > 5 → auto-hide comment
Notify moderators via Pusher

Moderator dashboard:
View flagged comments
Accept flag → hide/delete comment
Reject flag → restore visibility
Add official moderator reply

✅ Key Technical Advantages

Redis counter
 Fast and avoids multiple DB reads for flag counts.
 Helps auto-hide spammy or offensive comments dynamically.
Events & real-time notifications
 Using ShouldBroadcast ensures moderators get notified instantly.
 Multiple types of notifications can coexist:
   CommentFlagged → moderation
   ThreadCommented → thread author
   CommentReplied → parent author
MongoDB for threaded comments
 Flexible document-based storage makes hierarchical comment structures easier to manage.
 Fields like path and depth allow efficient sorting and querying.
Moderation logs
 Auditing: Every moderator action is tracked for transparency.
Scalable for multiple notification types
 Each notification is its own event type.
 Example: a single comment can trigger:
   Moderation flag notification.
   Notification to thread author.
   Notification to parent comment author.
 This separation avoids conflicts and keeps the system modular.
